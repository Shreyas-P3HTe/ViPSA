# ViPSA Testmaker Skill Composition Scheme

This document defines the **Level-3 skill layer** for ViPSA Testmaker.

The goal is to make every high-level test skill explicit enough that it can be:

1. generated by the Testmaker GUI,
2. exported as protocol-compatible CSV/JSON,
3. sequenced with other tests,
4. mapped cleanly onto the existing driver/orchestration stack.

The current Testmaker code already acts as an umbrella layer. It stores a `TEST_CATALOG`, assigns each test a category, status, mode, generator, protocol type, description, feasibility statement, and defaults. It then generates either sweep lists or pulse-slot lists and exports protocol JSON steps of the form:

```json
{"type": "DCIV", "params": {...}}
```

or

```json
{"type": "PULSE", "params": {...}}
```

It can optionally prepend `ALIGN` and `APPROACH` steps before the electrical step.

---

# 1. Layer Model

## Level 0 — Instrument/session layer

Responsible for direct instrument control.

Examples:

- VISA connection
- SCPI/TSP write/query
- output on/off
- reset/clear
- source/sense setup
- range/compliance setup
- switch matrix routing

## Level 1 — Driver primitives

The physical measurement atoms.

### Electrical primitives

- `hold_voltage_measure_current(...)`
- `source_voltage_measure_current(...)`
- `run_voltage_pulse_train(...)`

### Switch primitive

- `connect_named_route(...)`
- `open_all(...)`
- `close_channel(...)`
- `open_channel(...)`

## Level 2 — Orchestration blocks

These combine driver primitives into reusable protocol actions.

Examples:

- load CSV waveform
- generate voltage list
- split waveform into `p/n` or `pf/pb/nf/nb`
- insert read probe
- repeat cycles
- convert driver records to legacy GUI-compatible records
- export protocol JSON

## Level 3 — Skills

These are user-facing scientific tests.

A skill should answer:

- What is being measured?
- Why is it being measured?
- Which generator makes the waveform?
- Which protocol step is exported?
- Which driver primitive executes it?
- What parameters are required?
- What validation is needed?
- What output metrics should be computed later?

# 2. Testmaker Execution Model

## Sweep-mode tests

Sweep tests use the `dciv_bipolar` generator.

Typical flow:

```text
Testmaker skill
  -> get sweep parameters from GUI
  -> generate_voltage_data(...)
  -> save CSV: Time, Voltage
  -> export protocol:
       type = DCIV
       params = {
         sweep_path,
         pos_compl,
         neg_compl,
         sweep_delay,
         smu_select,
         use_4way_split,
         align=False,
         approach=False
       }
  -> Viewfinder / protocol runner executes DCIV
  -> SourceMeasureUnit.list_IV_sweep_split or list_IV_sweep_split_4
  -> driver.source_voltage_measure_current
```

## Pulse-mode tests

Pulse tests use one of the pulse-slot generators:

- `endurance`
- `bias_stress`
- `ltp_ltd`
- `ppf`
- `retention_read`

Typical flow:

```text
Testmaker skill
  -> get pulse parameters from GUI
  -> generate pulse slots
  -> save CSV: Time, Voltage
  -> export protocol:
       type = PULSE
       params = {
         pulse_path,
         compliance,
         pulse_width,
         smu_select,
         set_acquire_delay,
         align=False,
         approach=False
       }
  -> Viewfinder / protocol runner executes PULSE
  -> SourceMeasureUnit.pulsed_measurement / scan_read_vlist
  -> driver.run_voltage_pulse_train
```

## Info-mode tests

Info-mode tests are catalog entries without direct execution.

They are useful for:

- future feature planning
- documentation
- unsupported/manual workflows
- post-processing-only workflows

# 3. Common Composition Primitives

## `generate_voltage_list`

Creates a voltage/time waveform.

Used by:

- Current-Voltage Sweep
- Forming Sweep
- Device-to-Device Variability
- Temperature Stability
- Humidity Stability
- Temperature-Dependent I-V

## `generate_pulse_slots`

Creates a quantized pulse waveform.

Used by:

- Read Voltage Check
- Voltage Pulse Train
- Endurance Cycling
- Retention Readout Train
- Cycle-to-Cycle Variability
- Multi-Level Cell Programming
- Bias Stress
- LTP/LTD
- PPF
- Probability-Voltage Sigmoid

## `run_dciv_step`

Executes one exported `DCIV` protocol step.

Composition:

```text
load sweep CSV
select SMU
connect switch route
configure compliance
split sweep if requested
execute each sweep segment
optional read probe
return sweep records
```

## `run_pulse_step`

Executes one exported `PULSE` protocol step.

Composition:

```text
load pulse CSV
select SMU
connect switch route
configure compliance
configure pulse width/acquire delay
execute pulse train
return pulse records
```

## `insert_read_probe`

Adds low-voltage state readout.

Composition:

```text
use read voltage or tiny-IV profile
apply low-voltage read
measure current
compute resistance
append to read-probe records
```

## `loop_devices`

Runs the same skill across multiple devices.

Composition:

```text
for each device:
    route device
    optional align
    optional approach
    run skill
    save device result
```

## `loop_environment`

Runs the same skill across external condition points.

Composition:

```text
for each condition:
    wait for manual/external condition stabilization
    run electrical skill
    annotate condition metadata
```

## `loop_voltage_points`

Runs repeated pulse tests across a voltage range.

Composition:

```text
for each pulse voltage:
    generate/export one pulse waveform
    run repeated pulses
    compute switching probability
fit sigmoid after acquisition
```

# 4. Skill Specifications

## Basic Electrical

---

## 4.1 Current-Voltage (I-V) Sweep

### Description

Standard bipolar DC I-V sweep used to measure memristive switching behavior. It captures the current response during voltage ramps and is the main test for extracting `VSET`, `VRESET`, hysteresis, ON/OFF ratio, leakage, and switching polarity.

### Current Testmaker mapping

- Status: Ready Now
- Mode: Sweep
- Generator: `dciv_bipolar`
- Protocol type: `DCIV`
- Exported step: `{"type": "DCIV", "params": ...}`

### Composition

```text
Skill: Current-Voltage Sweep
  -> generate_voltage_data(dciv_bipolar)
  -> save sweep CSV
  -> export DCIV protocol step
  -> optional ALIGN
  -> optional APPROACH
  -> run_dciv_step
       -> load sweep CSV
       -> if use_4way_split:
              split into pf/pb/nf/nb
          else:
              split into p/n
       -> for each segment:
              choose compliance
              driver.source_voltage_measure_current(...)
       -> optional insert_read_probe
  -> save raw sweep records
  -> post-process VSET/VRESET/ON-OFF
```

### Driver primitives used

- `source_voltage_measure_current(...)`
- optionally `hold_voltage_measure_current(...)` for read probe

### Parameter fields

- `forward_voltage`
- `reset_voltage`
- `step_voltage`
- `timer_delay`
- `cycles`
- `forming_cycle`
- `forming_voltage`
- `peak_hold_steps`
- `return_to_zero`
- `sweep_mode`
- `pos_compl`
- `neg_compl`
- `use_4way_split`
- `smu_select`

### Defaults

```yaml
forward_voltage: 1.5
reset_voltage: -1.5
step_voltage: 0.05
timer_delay: 0.001
cycles: 2
forming_cycle: false
forming_voltage: 2.0
peak_hold_steps: 0
return_to_zero: true
pos_compl: 0.001
neg_compl: 0.01
use_4way_split: true
```

### Validation

- `forward_voltage > 0`
- `reset_voltage < 0`
- `step_voltage > 0`
- `timer_delay > 0`
- `cycles >= 1`
- `pos_compl > 0`
- `neg_compl > 0`
- if forming enabled: `forming_voltage >= forward_voltage`
- voltage must stay inside instrument and device safety limits
- compliance must stay below hard safety limit

### Outputs

- raw I-V trace
- current vs voltage
- time series
- compliance-hit markers
- VSET estimate
- VRESET estimate
- HRS/LRS resistance if read probe included
- ON/OFF ratio
- sweep success/failure flag

## 4.2 Forming / Electroforming Sweep

### Description

Initial high-voltage sweep used to create or activate a conductive filament in a pristine memristive device. It is more aggressive than a normal I-V sweep and should be compliance-limited to avoid hard breakdown.

### Current Testmaker mapping

- Status: Ready Now
- Mode: Sweep
- Generator: `dciv_bipolar`
- Protocol type: `DCIV`
- Exported step: `{"type": "DCIV", "params": ...}`
- Implemented as a higher first positive sweep followed by regular bipolar cycles.

### Composition

```text
Skill: Forming Sweep
  -> generate_voltage_data(forming_cycle=True)
  -> first cycle uses forming_voltage
  -> later cycles use forward_voltage
  -> save sweep CSV
  -> export DCIV protocol step
  -> run_dciv_step
       -> source_voltage_measure_current(...)
       -> monitor current/compliance
       -> if forming detected:
              mark forming event
  -> optional post-forming read probe
```

### Driver primitives used

- `source_voltage_measure_current(...)`
- optionally `hold_voltage_measure_current(...)`

### Parameter fields

- all I-V sweep fields
- `forming_cycle`
- `forming_voltage`
- forming compliance
- forming detection threshold, future extension
- stop-on-forming flag, future extension

### Defaults

```yaml
forward_voltage: 1.5
reset_voltage: -1.5
step_voltage: 0.05
timer_delay: 0.001
cycles: 2
forming_cycle: true
forming_voltage: 2.5
peak_hold_steps: 0
return_to_zero: true
pos_compl: 0.001
neg_compl: 0.01
use_4way_split: true
```

### Validation

- `forming_voltage > 0`
- `forming_voltage >= forward_voltage`
- forming compliance must be conservative
- stop if current exceeds safety limit
- stop if device becomes hard short
- forming sweep should generally run only once per pristine device

### Outputs

- forming I-V trace
- forming voltage estimate
- current jump marker
- post-forming resistance
- formed/not-formed flag
- breakdown flag

## 4.3 Read Voltage Check

### Description

Low-voltage non-destructive read test used to verify that the selected read voltage can sense the device state without disturbing it. This is important before retention, endurance, p-bit, and neuromorphic tests.

### Current Testmaker mapping

- Status: Ready Now
- Mode: Pulse
- Generator: `retention_read`
- Protocol type: `PULSE`
- Exported step: `{"type": "PULSE", "params": ...}`

### Composition

```text
Skill: Read Voltage Check
  -> generate_retention_read_slots
       -> optional write pulse
       -> repeated read pulses
       -> zero gaps
  -> save pulse CSV
  -> export PULSE protocol step
  -> run_pulse_step
       -> driver.run_voltage_pulse_train(...)
  -> evaluate read current/resistance drift
```

### Driver primitives used

- `run_voltage_pulse_train(...)`
- optionally `hold_voltage_measure_current(...)` for point-read implementation

### Parameter fields

- `read_voltage`
- `read_width`
- `read_gap`
- `read_pulses`
- `write_voltage`
- `write_width`
- `write_pulses`
- `pulse_compliance`
- `base_width`
- `set_acquire_delay`

### Defaults

```yaml
base_width: 0.001
write_voltage: 0.8
write_width: 0.002
write_gap: 0.001
write_pulses: 1
read_voltage: 0.1
read_width: 0.001
read_gap: 0.05
read_pulses: 20
erase_voltage: -0.8
erase_width: 0.002
erase_gap: 0.001
erase_pulses: 0
pulse_cycles: 1
cycle_gap: 0.0
initial_gap: 0.0
final_read_block: false
pulse_compliance: 0.001
set_acquire_delay: 0.0005
```

### Validation

- read voltage must be below known switching threshold
- `set_acquire_delay < base_width`
- read width must be resolvable by selected SMU
- repeated reads should not cause systematic resistance drift
- pulse list length must remain within backend limits

### Outputs

- read current train
- resistance train
- drift slope
- read-disturb flag
- recommended safe read voltage

## 5.1 Endurance Cycling

### Description

Repeated SET/READ/RESET/READ cycling test used to quantify switching durability. The main metric is how long the device maintains a clear ON/OFF memory window under repeated operation.

### Current Testmaker mapping

- Status: Ready Now
- Mode: Pulse
- Generator: `endurance`
- Protocol type: `PULSE`

### Composition

```text
Skill: Endurance Cycling
  -> generate_endurance_slots
       for each cycle:
          write pulse(s)
          read pulse(s)
          erase pulse(s)
          optional final read block
          cycle gap
  -> save pulse CSV
  -> export PULSE protocol step
  -> run_pulse_step
  -> parse sequence into cycles
  -> compute HRS/LRS per cycle
```

### Driver primitives used

- `run_voltage_pulse_train(...)`

### Parameter fields

- `write_voltage`
- `write_width`
- `write_pulses`
- `read_voltage`
- `read_width`
- `read_pulses`
- `erase_voltage`
- `erase_width`
- `erase_pulses`
- `pulse_cycles`
- `cycle_gap`
- `pulse_compliance`
- `set_acquire_delay`

### Defaults

```yaml
base_width: 0.001
write_voltage: 1.5
write_width: 0.001
write_gap: 0.001
write_pulses: 1
read_voltage: 0.1
read_width: 0.001
read_gap: 0.001
read_pulses: 1
erase_voltage: -1.8
erase_width: 0.001
erase_gap: 0.001
erase_pulses: 1
pulse_cycles: 100
cycle_gap: 0.0
initial_gap: 0.0
final_read_block: true
pulse_compliance: 0.01
set_acquire_delay: 0.0006
```

### Validation

- SET and RESET pulse polarities should be physically meaningful for the device stack
- read voltage must be non-disturbing
- pulse count must be manageable
- failure criteria should be defined before running:
  - ON/OFF below threshold
  - stuck ON
  - stuck OFF
  - excessive leakage
  - compliance hit every cycle

### Outputs

- cycle number
- LRS resistance
- HRS resistance
- ON/OFF ratio vs cycle
- endurance failure cycle
- switching success fraction

```yaml
base_width: 0.01

read_voltage: 0.1
read_width: 0.01
```

## 5.3 Cycle-to-Cycle Variability

### Description

Repeated switching on the same device to quantify stochastic variation between cycles. Useful for measuring distributions of switching probability, read resistance, ON/OFF ratio, and threshold-like behavior.

### Current Testmaker mapping

- Status: Partial / Manual
- Mode: Pulse
- Generator: `endurance`
- Protocol type: `PULSE`

### Composition

```text
Skill: Cycle-to-Cycle Variability
  -> generate_endurance_slots
       -> many repeated write/read/erase/read cycles
  -> run_pulse_step
  -> segment records by cycle
  -> compute distribution metrics
```

### Driver primitives used

- `run_voltage_pulse_train(...)`

### Parameter fields

Same as Endurance Cycling, with larger `pulse_cycles`.

### Defaults

```yaml
pulse_cycles: 200
write_voltage: 1.5
erase_voltage: -1.8
read_voltage: 0.1
pulse_compliance: 0.01
```

### Validation

- enough cycles required for meaningful statistics
- avoid excessive fatigue if the purpose is variability rather than endurance
- define valid cycle criteria

### Outputs

- distribution of LRS
- distribution of HRS
- distribution of ON/OFF
- switching success probability
- coefficient of variation
- cycle failure map

Runs the same electrical test across many devices to quantify spatial and device-to-device variability. This belongs to the batch/device orchestration layer more than the waveform generator itself.

### Composition

```text
Skill: Device-to-Device Variability
  -> generate one DCIV waveform
  -> for each device:
        route device
        optional ALIGN
        optional APPROACH
        run DCIV protocol
        save device result
  -> aggregate metrics across devices
```

### Driver primitives used

- `source_voltage_measure_current(...)`
- switch routing primitives
- optional contact-current probing

### Parameter fields

- all I-V sweep fields
- `device_list`
- `route_map`
- `align_before_each`
- `approach_before_each`

### Defaults


## 5.5 Multi-Level Cell Programming

### Description

Attempts to produce multiple stable conductance levels by applying controlled pulse sequences. Can be open-loop, using fixed pulse amplitudes/counts, or future closed-loop, where readback decides the next pulse.

### Current Testmaker mapping

- Status: Partial / Manual
- Mode: Pulse
- Generator: `ltp_ltd`
- Protocol type: `PULSE`

### Composition

```text
Skill: Multi-Level Cell Programming
  -> generate_ltp_ltd_slots
       -> repeated potentiation pulses with read after each
       -> repeated depression pulses with read after each
  -> run_pulse_step
  -> extract conductance after each read
  -> group conductance into levels
```

### Driver primitives used

- `run_voltage_pulse_train(...)`

### Parameter fields

- `write_voltage`
- `write_pulses`
- `erase_voltage`
- `erase_pulses`
- `read_voltage`
- `read_pulses`
- `pulse_compliance`
- future: `target_levels`, `tolerance`, `closed_loop`

### Defaults

```yaml
write_voltage: 1.0
write_pulses: 20
erase_voltage: -1.0
erase_pulses: 20
read_voltage: 0.1
read_pulses: 1
pulse_compliance: 0.005
```

### Validation

- conductance levels must be separable beyond noise
- read disturb must be checked

- retention of intermediate levels should be tested separately

### Outputs

- conductance vs pulse number
- candidate level centers
- level separation
- level overlap
- programming monotonicity

```text
Skill: Bias Stress
  -> generate_bias_stress_slots

       -> cycle gap

```

- `run_voltage_pulse_train(...)`

- `read_voltage`
- `read_width`

## 8.1 Potentiation / Depression (LTP/LTD)


### Driver primitives used

### Parameter fields

- `erase_width`
- `erase_pulses`

- Protocol type: `PULSE`

```text
  -> for each voltage:
        set write_voltage = current voltage
        generate_bias_stress_slots
        save pulse CSV
        export PULSE protocol JSON
  -> run each voltage protocol
  -> after acquisition:
        determine switch/no-switch events
        compute p(switch | V)
        fit sigmoid
```

### Driver primitives used

- `run_voltage_pulse_train(...)`

### Parameter fields

- `sigmoid_start_voltage`
- `sigmoid_stop_voltage`
- `sigmoid_step_voltage`

# 11. Suggested Code Cleanup Direction

## Keep Testmaker as the umbrella sequencer

Testmaker should own:

- catalog
- GUI defaults
- waveform generation
- protocol JSON export
- batch export helpers
- human-readable descriptions

## Do not put hardware execution in Testmaker

Hardware execution should remain in:

- protocol runner
- Viewfinder execution path
- `SourceMeasureUnit`
- driver files

## Add a formal `composition` field to each catalog entry

Example:

```python
"composition": {
    "generator_fn": "generate_endurance_slots",
    "export_type": "PULSE",
    "runner": "run_pulse_step",
    "driver_primitive": "run_voltage_pulse_train",
    "postprocess": ["parse_cycles", "compute_onoff_vs_cycle"],
}
```

## Add `outputs` and `validation` fields to each catalog entry

This makes the catalog useful for both GUI display and agentic execution.

Example:

```python
"outputs": [
    "raw_pulse_trace",
    "read_resistance_by_cycle",
    "onoff_by_cycle",
    "failure_cycle",
],
"validation": [
    "base_width > 0",
    "set_acquire_delay < base_width",

    "pulse_compliance > 0",
],
```
